# Contributing

Thanks for your interest in improving `Remote Deploy`.

This project is intentionally small and practical. The goal of these guidelines is not to create process for its own sake, but to reduce ambiguity around ownership, licensing, secrets, and review expectations.

## Before You Contribute

- Open an issue first for non-trivial changes so the scope and direction are clear
- Keep pull requests focused; avoid mixing unrelated refactors with the main change
- Update docs when behavior, setup, or safety expectations change
- Run `.\gradlew.bat buildPlugin` before opening a pull request

## What You May Submit

You may submit code, docs, tests, or build improvements that you wrote yourself or that you clearly have the right to contribute.

Please do **not** submit:

- copied proprietary code from an employer, client, or closed-source project
- third-party code, text, screenshots, icons, or other assets unless you have confirmed the license allows redistribution here
- secrets, tokens, passwords, private keys, certificates, or environment-specific credentials
- personal data, customer data, internal URLs, or other information you are not allowed to disclose
- code or content intended to bypass product licensing, DRM, access controls, or other legal restrictions

## Contribution Licensing

Unless we explicitly agree otherwise in writing, by submitting a contribution you agree that:

- you have the right to submit that contribution
- the contribution may be distributed under this repository's `MIT` license
- any required third-party notices or attributions have been included in the pull request

If you are unsure whether you can contribute something, stop and ask before opening the pull request.

## Third-Party Material

If your contribution includes any third-party material, you must explain it clearly in the pull request:

- what the material is
- where it came from
- what license applies
- what attribution or notice is required

If the answer is unclear, do not submit it yet.

## Security and Privacy

- Do not report sensitive vulnerabilities by posting exploit details, credentials, or private infrastructure data in a public issue
- Follow `SECURITY.md` for vulnerability reporting expectations
- If you discover accidentally committed secrets or personal data, do not spread them further in forks, issues, screenshots, or comments

## Review Expectations

Pull requests are reviewed for:

- scope and correctness
- maintainability
- compatibility with the existing plugin design
- documentation impact
- licensing, secrecy, and redistribution risk

## No Warranty or Support Promise

This repository is released under the `MIT` license and is maintained on a best-effort basis. Opening a pull request or issue does not create any service commitment, commercial relationship, or support obligation.
